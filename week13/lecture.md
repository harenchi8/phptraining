
# 第13週 座学テキスト

## 1. 最終課題①：要件定義 (11/15)

これまでの学習の集大成として、オリジナルのWebアプリケーションを企画・設計・開発します。今週はその最初のステップである「要件定義」と「設計」を行います。

### アプリケーションの企画

まず、「何を作るか」を決めます。これまでの学習内容（CRUD、認証、ファイルアップロードなど）を活かせるテーマを選びましょう。

**テーマの例:**
- **ブログアプリ**: Markdownで記事を投稿・編集できる。コメント機能、タグ付け機能。
- **タスク管理アプリ**: タスクの登録、状態（未着手、進行中、完了）の更新、期限設定、担当者割り当て。
- **簡易SNS**: テキストや画像を投稿でき、「いいね」やフォローができる。
- **商品レビューサイト**: 商品情報を登録し、ユーザーが5段階評価とレビューを投稿できる。
- **レシピ共有サイト**: レシピ（材料、手順、写真）を投稿・検索できる。

テーマを決めたら、そのアプリケーションの**目的**と**コア機能**を明確にします。「誰が、何のために使うのか？」「最も重要な機能は何か？」を考えましょう。

### 要件定義

要件定義とは、アプリケーションが満たすべき「要求」を具体的に定義する作業です。ここを曖昧にすると、開発の途中で手戻りが発生しやすくなります。

**1. 機能要件**
- アプリケーションが「何ができるか」を定義します。ユーザーの視点と、管理者の視点で機能を洗い出します。

**ユーザー機能の例（ブログアプリの場合）:**
- ユーザー登録ができる
- ログイン、ログアウトができる
- 記事を投稿、編集、削除できる（自分の記事のみ）
- 記事一覧を閲覧できる
- 記事をキーワードで検索できる
- 記事にコメントを投稿できる

**管理者機能の例:**
- 全てのユーザーを管理できる
- 不適切な投稿やコメントを削除できる

**2. 非機能要件**
- 機能面以外での要求を定義します。今回は簡易的なもので構いません。
- **ユーザビリティ**: スマートフォンでも快適に閲覧できる（レスポンシブデザイン）。
- **セキュリティ**: パスワードはハッシュ化して保存する。XSS, CSRF対策が施されている。
- **パフォーマンス**: ページ表示が遅すぎないこと。

## 2. 最終課題②：土台構築 (11/16)

### データベース設計 (ER図)

要件定義で洗い出した機能を元に、必要となるデータの構造を設計します。このとき、**ER図 (Entity-Relationship Diagram)** を使うと、テーブル（エンティティ）とその関係性（リレーションシップ）を視覚的に整理できます。

**ER図の構成要素:**
- **エンティティ (Entity)**: データのまとまり。通常はテーブルに相当します。（例: `users`, `posts`, `comments`）
- **アトリビュート (Attribute)**: エンティティが持つ情報。カラムに相当します。（例: `name`, `email`, `title`）
- **リレーションシップ (Relationship)**: エンティティ間の関係。「1対多」「多対多」など。

**ブログアプリのER図の例:**
```
+---------+       1..* +---------+       1..* +----------+
|  users  |------------|  posts  |------------| comments |
+---------+            +---------+            +----------+
| id (PK) |            | id (PK) |            | id (PK)  |
| name    |            | user_id |            | post_id  |
| email   |            | title   |            | user_id  |
| password|            | body    |            | body     |
+---------+            +---------+            +----------+
```
- `users`と`posts`は1対多（一人のユーザーが多くの投稿を持つ）。
- `posts`と`comments`も1対多（一つの投稿に多くのコメントが付く）。

### 画面設計 (ワイヤーフレーム)

アプリケーションの各画面のレイアウトや要素を設計します。手書きのスケッチや、Figma, Cacooなどのツールを使って簡単な**ワイヤーフレーム**を作成すると、完成イメージが具体的になります。

**作成する画面の例:**
- トップページ（記事一覧）
- 記事詳細ページ
- ログインページ
- ユーザー登録ページ
- 新規投稿ページ
- 投稿編集ページ

どこに何のボタンを置くか、フォームには何の項目が必要か、などを考えながら作成します。

### Laravelプロジェクトの土台構築

設計が終わったら、Laravelプロジェクトの土台を構築します。

1.  **新規Laravelプロジェクト作成**: `composer create-project ...`
2.  **Breezeのインストール**: `composer require laravel/breeze`, `php artisan breeze:install` ...
3.  **マイグレーションの作成**: ER図を元に、必要なテーブルのマイグレーションファイルを `php artisan make:model --migration` で作成します。
4.  **シーダー (Seeder) の作成**: 開発中に動作確認するための初期データ（ダミーデータ）を投入するシーダーを作成します。
    - `php artisan make:seeder UserSeeder` などでシーダーファイルを作成。
    - `database/seeders/DatabaseSeeder.php` から、作成したシーダーを呼び出すように設定します。
    - `php artisan db:seed` コマンドでデータを投入できます。
    - `php artisan migrate:fresh --seed` を使うと、全テーブルを削除→再作成→データ投入を一括で行えて便利です。

この週の目標は、**実際に動くコードを書き始める前段階の「設計」と「準備」を完了させること**です。ここをしっかり固めておくことで、来週からの実装フェーズがスムーズに進みます。
